# Skill Creator V2

## Prehook Gate (MANDATORY — fires before any skill logic)

Before doing ANY work, ask these 3 questions using AskUserQuestion in a SINGLE call. Do NOT proceed until all are answered. No skill logic executes until the prehook gate is complete.

### Prehook Questions

Use AskUserQuestion with all 3 questions:

**Question 1 — Intent:**
- header: "Intent"
- question: "What is the intent behind creating this skill? What problem are you trying to solve?"
- options:
  - label: "Automate a repetitive workflow", description: "A task you do repeatedly that should be codified into a reusable procedure"
  - label: "Enforce quality standards", description: "Conventions, patterns, or rules that should be followed consistently"
  - label: "Teach domain knowledge", description: "Specialized knowledge about APIs, schemas, business logic, or domain concepts"
  - label: "Integrate with external tools", description: "Working with specific tools, APIs, file formats, or services"

**Question 2 — Expected Output:**
- header: "Output"
- question: "What output should this skill produce when it runs?"
- options:
  - label: "Code / files generated", description: "The skill produces source code, config files, or project scaffolding"
  - label: "Structured text output", description: "Reports, analysis documents, formatted responses"
  - label: "Multi-step workflow", description: "A sequence of actions executed end-to-end"
  - label: "Analysis / recommendations", description: "Evaluations, decisions, or strategic guidance"

**Question 3 — Complexity Mode:**
- header: "Mode"
- question: "How thorough should the skill creation process be?"
- options:
  - label: "Simple (Recommended)", description: "Quick interview, build SKILL.md + assets, validate, done. Low token cost, fast turnaround."
  - label: "Advanced with benchmarking", description: "Full pipeline: deep interview, web research, build + test cases, A/B benchmark with isolated subagents, HTML comparison viewer, iterate. Token-heavy but battle-tested."

After all 3 answers are collected, branch to the appropriate pipeline below. The intent and output answers feed into BOTH pipelines — they inform interview questions, research direction, and eval criteria.

## Simple Pipeline

Use this when the user selected "Simple" in the prehook gate.

### Phase 1: Focused Interview

Based on the intent and output from prehooks, ask 1-2 targeted follow-up questions using AskUserQuestion:

- **Trigger phrases:** "What would a user say to activate this skill? Give 2-3 examples."
- **Scope:** "What should this skill NOT do? Any boundaries?"
- **Concrete example:** "Walk me through one complete usage scenario from start to finish."

One question per round, max 2 rounds. Use the prehook answers to skip questions that are already clear.

### Phase 2: Quick Research (conditional)

If the intent is "Integrate with external tools" OR the skill references specific APIs/libraries:
1. Run WebSearch for the relevant API documentation
2. Summarize key findings into a `references/` file in the skill being created
3. Skip this phase entirely if the skill is purely procedural

### Phase 3: Build

1. Read `references/skill-design-patterns.md` for structure guidance
2. Determine target directory — ask the user if unclear, default to `<project>/.claude/skills/`
3. Run `scripts/init_skill.py <skill-name> --path <target-directory>`
4. Write the SKILL.md for the new skill:
   - **Frontmatter:** name (hyphen-case) + comprehensive description (include ALL trigger phrases from interview)
   - **Body:** Instructions, workflow steps, decision trees as needed
   - **Example:** At least 1 concrete example showing: user prompt → skill behavior → expected output (MANDATORY)
   - Reference any scripts/references/assets created
5. Create any needed `scripts/`, `references/`, `assets/` files
6. Delete unused example files generated by init
7. Validate: `python3 scripts/quick_validate.py <skill-path>` — must pass

### Phase 4: Package

1. Run `python3 scripts/package_skill.py <skill-path>`
2. Report to user: skill name, file location, what it does, trigger phrases
3. Done.

## Advanced Benchmark Pipeline

Use this when the user selected "Advanced with benchmarking" in the prehook gate.

### Phase 1: Deep Interview

2-3 rounds of AskUserQuestion:
- **Round 1:** Purpose, scope, trigger phrases (informed by prehook intent/output answers)
- **Round 2:** Expected behavior walkthrough, edge cases, domain context
- **Round 3 (if needed):** Clarify ambiguities from previous rounds

### Phase 2: Research

1. Use WebSearch to find documentation for APIs, libraries, or domain knowledge relevant to the skill
2. Summarize findings into `references/` files for the skill being created
3. Skip if skill is purely procedural (no external dependencies)

### Phase 3: Build + Evals

1. Read `references/skill-design-patterns.md` for structure guidance
2. Determine target directory (ask user if unclear)
3. Run `scripts/init_skill.py <skill-name> --path <target-directory>`
4. Write SKILL.md for the new skill (same requirements as Simple Pipeline Phase 3)
5. Create `scripts/`, `references/`, `assets/` as needed
6. Read `references/eval-format.md` for test case format
7. Generate `evals/test-cases.md` inside the skill folder:
   - 3-5 test scenarios derived from interview answers
   - Each test case has: name, input prompt, expected behavior, pass/fail criteria
   - Eval rubric: correctness, completeness, quality, adherence (1-5 each)
8. Delete unused example files
9. Validate: `python3 scripts/quick_validate.py <skill-path>` — must pass

### Phase 4: A/B Benchmark

Read `references/subagent-prompts.md` for prompt templates.

1. Create temp directories:
```bash
mkdir -p /tmp/skill-benchmark-with /tmp/skill-benchmark-without
```

2. Read the generated SKILL.md contents and `evals/test-cases.md`
3. Record start timestamp
4. Spawn **two parallel subagents** using the Task tool (both in the SAME message):

**Agent A (with-skill):** `subagent_type: "general-purpose"`
- Prompt: inject full SKILL.md contents + all test case prompts
- Instruction: follow the skill exactly, write outputs to `/tmp/skill-benchmark-with/output-{n}.md`

**Agent B (without-skill):** `subagent_type: "general-purpose"`
- Prompt: inject only the test case prompts (NO skill loaded)
- Instruction: do best with general knowledge, write outputs to `/tmp/skill-benchmark-without/output-{n}.md`

5. Wait for both agents to complete
6. Record end timestamps for each
7. Measure output file sizes (character count) as token proxy

### Phase 5: Grade

1. Read all output files from both agents (`/tmp/skill-benchmark-with/` and `/tmp/skill-benchmark-without/`)
2. Read the eval rubric from `evals/test-cases.md`
3. For each test case, score BOTH agents' outputs on 4 dimensions (1-5 each):
   - **Correctness:** Did it produce the right result?
   - **Completeness:** Did it cover all aspects of expected behavior?
   - **Quality:** Is the output well-structured and polished?
   - **Adherence:** Did it follow the expected behavior / skill instructions?
4. Calculate per-test totals (average of 4 dimensions)
5. Calculate aggregates:
   - Average score: with-skill vs without-skill
   - Win rate: how many tests with-skill scored higher
   - Time delta: difference in completion time
   - Output size delta: difference in output chars

### Phase 6: HTML Review

1. Assemble results into JSON:
```json
{
  "skill_name": "<skill-name>",
  "metrics": {
    "with_skill": {"avg_score": N, "time_seconds": N, "output_chars": N},
    "without_skill": {"avg_score": N, "time_seconds": N, "output_chars": N},
    "win_rate": "X/Y"
  },
  "tests": [
    {
      "name": "Test N: description",
      "prompt": "the test prompt",
      "with_skill": {"output": "...", "scores": {"correctness": N, "completeness": N, "quality": N, "adherence": N}, "total": N, "pass": true/false},
      "without_skill": {"output": "...", "scores": {...}, "total": N, "pass": true/false}
    }
  ]
}
```
2. Write JSON to `/tmp/skill-benchmark-results.json`
3. Run: `python3 scripts/generate_comparison.py /tmp/skill-benchmark-results.json`
4. Run: `python3 scripts/open_viewer.py /tmp/skill-benchmark-results.html`
5. Prompt user: **"Review the benchmark comparison in your browser. What feedback do you have? Reply 'looks good' to finalize, or describe what to change."**

### Phase 7: Iterate

- **If user has feedback:**
  1. Modify the skill's SKILL.md / scripts / references based on feedback
  2. Re-run Phases 4-6 only (benchmark → grade → review)
  3. Repeat until user says "looks good" or has no more feedback
- **If user is satisfied:**
  1. Run `python3 scripts/package_skill.py <skill-path>`
  2. Report: skill name, location, final benchmark scores, what it does
  3. Done.

## Examples

### Example 1: Simple Mode

**User says:** "Create a skill for formatting Python code with Black"

**Prehook answers:**
- Intent: Automate a repetitive workflow
- Output: Code / files generated
- Mode: Simple

**Skill behavior:**
1. Focused interview: asks about trigger phrases ("format this", "run black"), scope (just formatting, not linting)
2. Quick research: WebSearch for Black CLI docs, config options
3. Build: creates `black-formatter/SKILL.md` with CLI commands, pyproject.toml config patterns, concrete example
4. Package: validates, packages, reports location
5. Done in ~2 minutes

### Example 2: Advanced Mode

**User says:** "Create a skill for generating API documentation from OpenAPI specs"

**Prehook answers:**
- Intent: Teach domain knowledge
- Output: Structured text output
- Mode: Advanced with benchmarking

**Skill behavior:**
1. Deep interview (3 rounds): OpenAPI version support, output format preferences, template style, edge cases (webhooks, auth schemes)
2. Research: WebSearch for OpenAPI 3.1 spec, common patterns, best doc generators
3. Build + Evals: creates SKILL.md + `references/openapi-guide.md` + 4 test cases covering different API shapes
4. A/B benchmark: Agent A (with skill) vs Agent B (without) execute all 4 test cases in parallel
5. Grade: with-skill averages 4.5/5, without-skill averages 2.1/5
6. HTML viewer opens in browser — side-by-side comparison of all outputs with scores
7. User feedback: "Add support for webhooks section" → skill updated → re-benchmark → scores improve
8. User satisfied → package → done

### Example 3: Simple Mode — Domain Knowledge

**User says:** "Make a skill that knows our company's database schema"

**Prehook answers:**
- Intent: Teach domain knowledge
- Output: Analysis / recommendations
- Mode: Simple

**Skill behavior:**
1. Focused interview: asks for schema details, common queries, which tables matter most
2. No research needed (internal knowledge)
3. Build: creates `company-schema/SKILL.md` with `references/schema.md` containing table definitions, relationships, common query patterns
4. Package and done
